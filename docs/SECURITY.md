# Security posture (SPEC section 11, Milestone 7)

This document is the honest record. Statuses below are self-assessed at
demo scope against real scan output, not certified by an external audit,
and every "partial" names its gap. Live counterpart: the Security
Posture screen, fed by `GET /api/security/posture`, which relays
`infra/scans/out/summary.json` unmodified.

## 1. Scan harness

`./infra/scans/run_all.sh` runs every scanner that is installed and
records absent ones as `{"status": "not_installed"}`. Raw artifacts land
in `infra/scans/out/<tool>.json`; `infra/scans/summarize.py` writes
`summary.json` (`jq .summary infra/scans/out/summary.json`, SPEC 17
step 6). Severity mapping honesty:

- pip-audit emits no severity field: findings count toward
  `findings_total` only, with a note saying so.
- gitleaks findings all count as critical (a leaked secret has no
  lesser grade); reports are redacted.
- npm audit `moderate` maps to `medium`, `info` to `low`; trivy and
  grype `UNKNOWN` is reported as `unknown`, never dropped.

Scanner versions for the recorded 2026-07-12 run: pip-audit 2.10.1,
bandit 1.9.4, semgrep 1.169.0, gitleaks 8.30.1, trivy 0.71.1,
grype 0.114.0, npm 11.12.1 (bundled audit).

Result of that run: zero criticals in repo sources and dependencies
(pip-audit, npm audit, bandit, semgrep, gitleaks, trivy fs, grype).
Three criticals exist in the api container image's Debian base packages
with no upstream fix published; they are accepted risks (section 6)
because no pin bump can remove them.

Gitleaks allowlist (`infra/scans/gitleaks.toml`), each entry reasoned:
`.env` is the designated runtime secret store (generated by
`scripts/bootstrap_env.sh`, chmod 600, gitignored, must contain
secrets); `node_modules/` and `frontend/dist/` are third-party or
derived content; `infra/scans/out/` is the reports themselves. CI runs
gitleaks unallowlisted over the git history as well.

## 4. Threat model

In scope: a single-host docker compose deployment on the demo Mac, one
investigator team, synthetic data. Assets: the audit ledger's
integrity, credentials in `.env`, the evidence packet outputs.

- **Tampering with the ledger.** SPEC 8.4, stated verbatim on both the
  Custody and Security screens: a hash chain is tamper-evident, not
  immutable; an attacker privileged enough to rewrite a record and
  recompute every downstream hash defeats internal verification, so the
  chain head is anchored out of band on every exported packet, in the
  structured log, and as a Prometheus metric. Below that privilege
  level, UPDATE/DELETE/TRUNCATE are rejected by SECURITY DEFINER
  triggers and the app role cannot alter rows at all.
- **Exfiltration / silent egress.** Docker cannot both publish ports
  and block WAN on the same network, so zero egress is measured, not
  asserted (SPEC 2): the posture endpoint performs a live outbound TCP
  attempt (1.1.1.1:443, 2 s timeout) and lists the complete configured
  outbound endpoint set, which is exactly one HTTP endpoint (host-local
  Ollama) plus the internal DB DSN. The demo runs with Wi-Fi off; the
  self-test then shows blocked, live, in front of the judge.
- **Credential theft on the host.** `.env` is chmod 600, gitignored,
  gitleaks-scanned repo-wide; host disk is FileVault-encrypted. A root
  attacker on the demo Mac is out of scope and said so.
- **Prompt/output manipulation of the language layer.** The generative
  step cannot introduce claims: the F9 gate withholds any claim not
  verbatim-grounded in the corpus, and the planted-fabrication demo
  proves the block live. The deterministic core never consumes LLM
  output as input.
- **Dependency compromise.** Pinned versions plus the scanner suite;
  residual risk in section 6.

## 5. Encryption posture and key custody (honest)

- **In transit, demo:** plain HTTP on localhost only (published ports
  8090/8081/3001 bind on the demo machine). No TLS claim is made
  anywhere in the UI; the posture panel says this outright.
- **In transit, pilot path:** nginx TLS termination with
  department-issued certificates in front of `web` and `api`; documented
  procedure, not yet deployed, therefore not claimed.
- **At rest:** FileVault full-disk encryption on the demo Mac covers the
  Postgres volume and the offline bundle. Postgres-level encryption
  (pgcrypto column level or TDE) is a pilot decision, not a demo claim.
- **Key custody:** all secrets (JWT secret, DB passwords, seeded user
  passwords, Grafana admin) are generated by `scripts/bootstrap_env.sh`
  into `.env` (chmod 600, gitignored, printed once at generation). No
  secret is committed, baked into an image, or logged; gitleaks enforces
  the repo side in CI over full git history. Rotation = rerun bootstrap
  and recreate the stack; there is no KMS/HSM at demo scale and this
  document does not pretend otherwise.

## 7. CI wiring

`.github/workflows/ci.yml` runs, alongside the existing test, build,
truth-rule, pip-audit, and gitleaks jobs: bandit (fails on high or
above), semgrep (`--config auto`, fails on any finding), and trivy
(repo filesystem scan, exit code 1 on CRITICAL). The offline
equivalent and demo-day source of truth is `./infra/scans/run_all.sh`.
